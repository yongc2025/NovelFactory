# 系统设计

## 目标

把长篇小说创作变成可追踪、可续写、可审校、可回滚的项目工程。

系统关注的不是单次生成多少正文，而是每一次生成之后，项目状态是否被正确保存。

## 三层结构

```text
第一层：创作方法
负责命题、人物、世界观、章节、场景、正文、审校。

第二层：项目状态
负责保存项目圣经、角色手册、章节表、伏笔表、认知状态、情绪线、修订记录。

第三层：版本管理
由 GitHub 保存每次修改，便于查看差异、回滚、分支试验。
```

## 标准工作循环

```text
读取状态
→ 判断阶段
→ 生成本轮内容
→ 检查连续性
→ 更新台账
→ 写入交接包
→ 提交版本
```

## 最小可用状态

开始正式写正文前，至少需要：

1. 项目圣经
2. 十二层创作台账
3. 角色手册
4. 章节表
5. 人物认知状态
6. 当前章节交接包

如果资料不完整，应先根据已有内容恢复临时状态，并把不确定内容标记为“待确认”。

## 修改分类

每一次修改先分类，再执行：

1. 文风润色：只影响正文和审校报告。
2. 章节结构调整：影响章节卡、场景卡、正文、章节表。
3. 设定更新：影响项目圣经、世界观、角色手册、连续性台账。
4. 追溯调整：影响已写章节、信息释放表、伏笔台账、人物认知状态、修订记录。

## 分支建议

```text

revision/项目编号-主题：重大修改分支
experiment/项目编号-主题：试验剧情分支
```
