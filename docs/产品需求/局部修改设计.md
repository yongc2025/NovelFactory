# 局部修改设计

> 版本：1.0 | 更新日期：2026-06-12
> 本文档定义场景级、段落级局部修改的设计方案。

---

## 1. 问题定义

### 1.1 当前问题

**现状**：只能整章重新生成或手动编辑整章

**问题**：
- 第 3 章第 2 场写得不好 → 只能重新生成整章
- 改了一段对话 → 要重新等整章生成
- 无法保留满意的部分，只改不满意的部分

### 1.2 作家真实需求

```
作家需要的操作粒度：

场景级：重新生成第3章第2场，保留其他场
段落级：只改第5段对话，保留其他段落
句子级：只润色这一句话
```

---

## 2. 设计目标

### 2.1 核心目标

1. **场景级操作**：重新生成单个场景
2. **段落级操作**：AI 润色/扩写单个段落
3. **保留上下文**：局部修改时保持整体连贯

### 2.2 操作粒度

| 粒度 | 操作 | 影响范围 |
|------|------|----------|
| 全章 | 重新生成整章 | 整章替换 |
| 场景 | 重新生成单场景 | 场景替换 |
| 段落 | AI 润色/扩写 | 段落替换 |
| 句子 | AI 改写 | 句子替换 |

## 4. 上下文管理

### 4.1 局部修改的上下文

**问题**：局部修改时，如何保持整体连贯？

**方案**：提供足够的上下文

```
局部修改时的上下文：

1. 全局上下文
   - 角色信息
   - 世界观设定
   - 当前章节大纲

2. 局部上下文
   - 前文摘要
   - 前一场景/段落结尾
   - 后一场景/段落开头

3. 当前内容
   - 要修改的内容
   - 内容的元数据（场景号、段落号）
```

### 4.2 修改后的上下文更新

**问题**：局部修改后，后文的上下文可能不一致

**方案**：

```
局部修改后：

1. 更新当前内容
2. 生成新的摘要（如果影响较大）
3. 标记后文可能受影响的部分
4. 提示用户检查后文连贯性
```

## 5. 技术实现

### 5.1 数据模型

```python
# 场景数据结构
class Scene:
    id: str
    chapter_num: int
    scene_num: int
    location: str
    content: str
    summary: str  # 场景摘要，用于上下文传递
    word_count: int
    status: str

# 段落数据结构（可选，如果需要段落级追踪）
class Paragraph:
    id: str
    scene_id: str
    paragraph_num: int
    content: str
    type: str  # narrative/dialogue/description
```

### 5.2 API 接口

| 方法 | 端点 | 描述 |
|------|------|------|
| POST | /api/projects/{id}/scenes/{num}/regenerate | 重新生成场景 |
| PUT | /api/projects/{id}/scenes/{num} | 编辑场景 |
| POST | /api/ai/polish | AI 润色 |
| POST | /api/ai/expand | AI 扩写 |

### 5.3 请求/响应格式

```json
// POST /api/projects/{id}/scenes/{num}/regenerate
{
  "project_id": "xxx",
  "chapter_num": 3,
  "scene_num": 2,
  "context": {
    "prev_summary": "...",
    "prev_scene_ending": "...",
    "characters": [...],
    "scene_outline": {...}
  }
}

// Response
{
  "scene": {
    "id": "new_scene_id",
    "content": "新生成的场景内容...",
    "word_count": 500,
    "status": "draft"
  },
  "changes_summary": "重新生成了场景，主要修改了对话部分"
}
```
