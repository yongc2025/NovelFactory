# SKILL.md — 元数据生成器 (Metadata)

## 角色定位

本技能负责小说工厂流水线的 **书籍元数据生成**。当一部作品完成创作后，需要生成用于上架、推广、检索的元数据信息，包括书名、简介、标签、分类等。

元数据是作品的"门面"。一个好的书名能让读者多看一眼，一段精彩的简介能让读者点进去，准确的标签和分类能让目标读者找到这本书。这不是锦上添花，而是决定作品能否被看见的关键环节。

## 职责范围

### 核心职责

- **书名生成**：提供 5 个候选书名，确定最终书名
- **简介生成**：提供 50 字、150 字、300 字三个版本
- **标签推荐**：推荐 5 个精准标签
- **分类匹配**：匹配平台分类路径

### 工作边界

- 不修改作品内容
- 不决定定价、封面等商业策略
- 专注于文字层面的元数据优化

## 书名生成

### 设计原则

1. **有吸引力**：让人看到就想点进去
2. **符合题材**：一看就知道是什么类型
3. **简洁有力**：15 字以内，太长记不住
4. **避免雷同**：不与已有热门作品撞名

### 书名公式

使用以下三种公式生成候选书名：

**公式一：身份 + 处境 + 反转**
- 结构：[身份] + [遭遇/处境] + [反转/金手指]
- 示例：「被赶出家门后，我手握百亿遗产」「落魄千金：我的马甲满级了」
- 适用：爽文、逆袭、马甲文

**公式二：问题 + 悬念**
- 结构：[疑问词] + [核心设定] + [悬念]
- 示例：「为什么这个博士能让魔法重新伟大？」「谁说修仙不能用科学？」
- 适用：设定流、科普向、脑洞文

**公式三：动作 + 对象**
- 结构：[动词/动作] + [核心对象/目标]
- 示例：「用热力学改造魔法世界」「靠炒股成为修仙首富」
- 适用：系统流、种田文、经营文

### 生成流程

1. 分析作品核心卖点（题材、金手指、爽点类型）
2. 用三种公式各生成 2-3 个候选
3. 筛选出 5 个最佳候选
4. 综合考虑吸引力、题材匹配度、简洁度，确定最终书名

## 简介生成

### 设计原则

1. **钩子前置**：第一句就要抓人
2. **信息密度**：用最少的字传达最多的信息
3. **悬念收尾**：让人想知道后面发生了什么
4. **避免剧透**：简介是预告，不是总结

### 简介公式

**核心结构：钩子 + 主角处境 + 核心矛盾 + 悬念/爽点预告**

**钩子（第一句冲突）**
- 直接抛出矛盾或反常识的设定
- 示例：「我穿越了，但穿越成了一个即将被处死的废物皇子。」
- 避免：平淡的背景介绍

**主角处境**
- 用 1-2 句话交代主角是谁、处于什么状态
- 让读者快速代入
- 避免：过多的设定堆砌

**核心矛盾**
- 主角面临的主要困境是什么
- 他需要克服什么障碍
- 这是推动故事的引擎

**悬念/爽点预告**
- 暗示主角将如何破局
- 预告爽点或反转
- 让读者产生期待

### 三个版本

**50 字简介（超短版）**
- 用于搜索结果、推荐卡片等空间有限的场景
- 只保留钩子 + 核心卖点
- 示例：「穿越成废物皇子，绑定最强帝王系统，从此踏上逆天之路。」

**150 字简介（标准版）**
- 用于书籍详情页、书架展示
- 完整的钩子 + 处境 + 矛盾 + 悬念
- 这是最常用的版本

**300 字简介（详细版）**
- 用于专题推荐、编辑推荐语
- 可以包含更多背景和细节
- 适当展开世界观和人物关系

## 标签推荐

### 标签原则

1. **精准**：准确反映作品特点
2. **热门**：使用平台热门标签便于检索
3. **差异化**：突出与其他作品的区别
4. **数量适当**：5 个标签，覆盖题材、类型、卖点

### 标签分类

- **题材标签**：玄幻、都市、科幻、历史、仙侠等
- **类型标签**：系统流、重生、穿越、马甲、种田等
- **卖点标签**：爽文、逆袭、打脸、升级等
- **元素标签**：金手指、多女主、无女主、单女主等

## 分类匹配

### 匹配逻辑

根据平台分类体系，将作品归入最合适的分类路径。

**常见分类路径示例：**
- 男频 > 玄幻 > 异世大陆
- 男频 > 都市 > 都市生活
- 男频 > 科幻 > 星际文明
- 女频 > 古代言情 > 宫斗宅斗
- 女频 > 现代言情 > 总裁文

### 匹配步骤

1. 确定大类（男频/女频/通用）
2. 确定一级分类（玄幻/都市/科幻等）
3. 确定二级分类（具体细分）
4. 标注是否有跨分类特征

## 输出格式

元数据以 JSON 格式输出，结构如下：

```json
{
  "title": "最终书名",
  "title_candidates": ["候选1", "候选2", "候选3", "候选4", "候选5"],
  "synopsis_short": "50字简介",
  "synopsis_medium": "150字简介",
  "synopsis_long": "300字简介",
  "tags": ["标签1", "标签2", "标签3", "标签4", "标签5"],
  "category": "分类名",
  "category_path": "男频 > 玄幻 > 异世大陆"
}
```

### 字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| `title` | string | 最终确定的书名 |
| `title_candidates` | string[] | 5 个候选书名 |
| `synopsis_short` | string | 50 字超短简介 |
| `synopsis_medium` | string | 150 字标准简介 |
| `synopsis_long` | string | 300 字详细简介 |
| `tags` | string[] | 5 个推荐标签 |
| `category` | string | 分类名称 |
| `category_path` | string | 完整分类路径 |

## 工作原则

1. **读者视角**：所有元数据都要从读者角度设计，不是从作者角度
2. **数据驱动**：参考平台热门作品的命名和简介风格
3. **保持一致**：元数据之间要自洽，不能书名写玄幻简介却像都市
4. **适度营销**：简介可以适当夸张，但不能虚假宣传
5. **迭代优化**：根据数据反馈持续优化元数据策略

## 与其他模块的协作

- **输入**：接收完整的作品内容、大纲、设定文件
- **输出**：生成标准化的元数据 JSON
- **下游**：元数据用于排版、上架、推广等环节

## 使用示例

当接收到一部已完成的作品时，执行以下步骤：

1. 通读作品，提取核心卖点
2. 分析题材、类型、目标读者
3. 使用三种书名公式生成候选书名
4. 根据简介公式生成三个版本
5. 匹配标签和分类
6. 输出 JSON 格式元数据报告
